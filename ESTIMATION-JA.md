# SMG IPGO+OPE Hub 工数概算 

## 概要

本見積もりは、IPGOおよびOPEプラットフォームに対する要求変更をカバーするものです。
お読みいただく前に、現在のIPGOおよびOPEシステムは、IPGOとOPE間のこのような高度で複雑な統合や、機密性の高い多数の新機能を扱うようには設計されていない点にご留意ください。完全なカスタムバックエンドの再構築（リライト）が必要になる可能性が非常に高いです。Directusで対応できる可能性もありますが、長期的には開発速度が低下し、管理画面UIの柔軟性が損なわれる恐れがあります。潜在的なバックエンドの再構築に備え、1〜1.5ヶ月のバッファ（予備期間）を見込んでください。

## 数値の前提条件

- **チーム構成:** 社内開発者2名 ＋ 随時軽微な外部委託
- **開発手法:** すべての機能は、既存の安定したコードベース上の小規模で独立した追加機能として扱う
- **技術的負債:** なし（ただし、OPEのSpringbootには一部存在すると予想される）
- **セキュリティ:** 本格的なセキュリティ監査、ペネトレーションテスト、負荷テストは行わない
- **法務:** 見積もりには正式な法務確認プロセスは含まれていません
- **デザイン:** 正式なUX/UIデザインフェーズはなし。すべて既存のコンポーネントを使用する
- **外部連携:** 多言語、多通貨、決済ゲートウェイ、電子署名APIなどは、最小限の労力で「問題なく動作する」ものとする
- **イベント連携:** ワンダーフェスティバル / EXPOアプリ（Spring Boot + React Native）との統合は順調に進むものとする
- **クライアント:** クライアントからのフィードバックは即座に行われ、要件変更は発生しない
- **その他:** 休日なし、スコープクリープ（仕様肥大化）なし、デバッグに数日かかるようなバグはゼロとする

#### ※これらは、クライアントが機能を選択できるように提示するための「ベストケース（最短・最小）」の数値です。

### 制限事項と潜在的な課題

IPGOプラットフォームの見積もりは楽観的なロードマップを提示していますが、納品、拡張性、成功に影響を与えうるいくつかの制限とリスクが存在します。これらは、本書の野心的なスコープ、現在の技術スタック（Directus/Nuxt/Spring Boot/React Native）、および少人数チームの制約に基づいています。

| 課題カテゴリー | 説明 | 潜在的な影響 | 緩和策の提案 |
|----------------|-------------|------------------|-----------------------|
| **工数の過小評価** | 見積もりは楽観的であり、技術的負債や変更がないことを前提としています。現実のバグ、統合、スコープ拡大により、期間は2〜3倍に延びる可能性があります。 | ローンチの遅延（例：WF 2026への準備不足）、予算超過。 | デリバリーを厳格に段階分けする。成果物を小さく保つ。 |
| **Directusの制限** | Directusはデータ/認証の処理には優れていますが、複雑なフロー（ロイヤリティ、AIマッチング、リアルタイムWF同期）には大幅な拡張が必要です。規模が拡大すると、DBパフォーマンスやカスタムロジックがボトルネックになる可能性があります。 | メンテナンスの負債。WFのような高トラフィックイベント（5万人以上のユーザー）時のパフォーマンス問題。 | 拡張機能の複雑さを監視する。コードの50%以上がカスタムになる場合は、カスタムバックエンドへ移行する（1〜2ヶ月の工数）。 |
| **チームとリソースの制約** | 開発者2名（うち1名はジュニア）のみ。専任のQA、デザイナー、弁護士が不在。外部委託は品質やコミュニケーションミスのリスクがあります。 | チームの疲弊、進捗の遅れ。不完全なテストによる本番環境でのバグ。 | QAを早期に雇用または外部委託する。 |
| **法的およびコンプライアンスのリスク** | 契約、ロイヤリティ、紛争、日本特有の証書などの機能には徹底的なレビューが必要です。不備があればIP法（例：日本のJASRACなど）に違反する可能性があります。 | 訴訟、プラットフォームの停止、監査による遅延。 | 機能ごとに外部弁護士を関与させる（各1〜3ヶ月追加）。JP/KRのコンプライアンスを優先する。 |
| **スケーリングとパフォーマンス** | リアルタイム機能（同期、通知、WFでのAR）は大規模環境で未テストです。本番環境の要素（負荷テスト、バックアップ）が欠如していると、高負荷時にシステムがダウンする可能性があります。 | イベント中のダウンタイム、データ損失。 | 小規模なWF類似イベントでパイロットを実施する。監視ツールを早期に導入する。 |
| **統合の依存関係** | 外部API（DocuSign、決済、AIサービス）への依存はダウンタイムのリスクをもたらします。WF/EXPOアプリの同期はアップデートにより破損する可能性があります。 | 機能不全、サポート需要の増加。 | フォールバック（代替）メカニズムを用意する。APIのバージョン管理を厳格に行う。 |

## 結論

概要で述べたIPGOプラットフォームは、IP管理、クリエイター支援、ファンエンゲージメント、ワンダーフェスティバル統合を行う野心的な統合システムです。しかし、そのスコープの複雑さ、ロイヤリティやAIマッチングなどのワークフローに対するDirectusの大幅な拡張が必要となるため、2名の開発チームで完全な構築を行うことの実現可能性は低いです。データ操作の規模が拡大するにつれ、事実上のカスタムバックエンド化していく可能性があります。

技術的負債や未テストの統合によるチームの疲弊、納品遅延、メンテナンスの問題といったリスクがあります。並行作業、法務レビュー、セキュリティ監査などの本番環境に不可欠な要素に対応する能力が小規模チームには不足しているため、大幅な外部委託や採用なしに全48機能のビジョンを実現するのは非現実的です。

目標を小さく保ち、大きな塊ではなく、モジュール化されたリリース可能な機能（例：WF承認機能と基本機能から開始など）に焦点を当て、継続的にリリースしていくのであれば実現可能です。長期間のスケーラビリティを確保するために、Directusがリアルタイム処理やAI運用で限界に達した場合、プロジェクト途中でカスタムバックエンドへの移行など、システムの根本的な変更が必要になる可能性があります。海洋堂（Kaiyodo）からの反復的なフィードバックを優先し、現実的な制約に合わせて持続可能な進捗を目指してください。
